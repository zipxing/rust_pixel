# Why We Ditched SDF/MSDF for Mipmaps in Our Pixel Engine
# 为什么我们在像素引擎中放弃了 SDF/MSDF，改用 Mipmap

---

*This post is about [RustPixel](https://github.com/zipxing/rust_pixel), a Rust 2D game engine that supports terminal, desktop (wgpu), and web (WebGPU/WebGL2) rendering from a single codebase. One of its apps, MDPT, is a fullscreen Markdown presentation tool that needs crisp text at any resolution — which is how we ended up on this journey.*

*本文讲的是 [RustPixel](https://github.com/zipxing/rust_pixel)，一个 Rust 2D 游戏引擎，用一套代码支持终端、桌面（wgpu）和 Web（WebGPU/WebGL2）渲染。其中一个应用 MDPT 是全屏 Markdown 演示工具，需要在任意分辨率下保持文字清晰——这就是我们踏上这段旅程的原因。*

---

## The Problem / 问题

RustPixel renders everything — sprites, TUI characters, CJK text, emoji — from a single texture atlas using instanced rendering (one draw call). When we built MDPT (Markdown Presentation Tool), we hit a wall: a TUI cell designed at 32×64 pixels in the atlas gets displayed at 4–8× magnification on a 4K fullscreen display. Pure bitmap textures turn into a blurry mess.

RustPixel 用一张纹理图集 + 实例化渲染（单次 draw call）绘制所有内容——精灵、TUI 字符、中文、Emoji。当我们开发 MDPT（Markdown 演示工具）时碰壁了：图集里 32×64 像素的 TUI 字符格子，在 4K 全屏上要放大 4-8 倍，纯位图纹理直接糊成一片。

We needed resolution-independent text rendering. The obvious answer? SDF.

我们需要分辨率无关的文字渲染。显而易见的方案？SDF。

---

## The SDF/MSDF Era (v2.1 – v2.3) / SDF/MSDF 时代（v2.1 – v2.3）

We went all-in on distance fields:

我们全力投入距离场方案：

- **TUI characters** (ASCII, box-drawing, Nerd Font icons): True MSDF generated by `msdfgen`, preserving sharp corners
- **CJK characters** (PingFang SC): Bitmap-to-SDF via `scipy.ndimage.distance_transform_edt` (Apple's PingFang.ttc doesn't expose glyph outlines, so `msdfgen` can't process it)
- **Sprites & Emoji**: Stayed as regular bitmaps

- **TUI 字符**（ASCII、制表符、Nerd Font 图标）：用 `msdfgen` 生成真 MSDF，保留尖角
- **CJK 字符**（苹方字体）：位图转 SDF，用 `scipy.ndimage.distance_transform_edt`（苹果的 PingFang.ttc 不暴露字形轮廓，`msdfgen` 处理不了）
- **精灵 & Emoji**：保持普通位图

The 8192×8192 atlas had four rendering regions, and the fragment shader had a dual path — MSDF distance field rendering for text, regular bitmap sampling for everything else. We encoded the MSDF flag in the sign bit of `origin_y` (zero extra memory per instance):

8192×8192 的图集有四个渲染区域，片段着色器分双路径——文字用 MSDF 距离场渲染，其他用普通位图采样。我们把 MSDF 标记编码在 `origin_y` 的符号位里（每个实例零额外内存）：

```wgsl
// Vertex shader extracts the flag
output.v_msdf = select(0.0, 1.0, instance.a1.y < 0.0);
let origin = abs(instance.a1.xy);

// Fragment shader: MSDF path
let d = median3(texColor.r, texColor.g, texColor.b);
let w = max(fwidth(d), 0.03);
let alpha = smoothstep(0.5 - w, 0.5 + w, d);
```

And it worked! Text was crisp at any zoom. Ship it, right?

效果不错！文字在任何缩放下都清晰。可以发布了吧？

---

## Why We Abandoned It / 为什么我们放弃了

Not so fast. After living with SDF for several months, the pain points accumulated:

没那么快。跟 SDF 共处几个月后，痛点逐渐积累：

### 1. Two Rendering Pipelines = Double the Bugs / 两套渲染管线 = 双倍 Bug

The shader had to branch on every fragment: is this MSDF or bitmap? Bold text needed two completely different implementations — MSDF bold lowers the threshold from 0.5 to 0.45 to expand the glyph, bitmap bold samples neighboring texels. Every rendering feature had to be implemented twice and tested twice.

着色器每个片段都要判断：这是 MSDF 还是位图？粗体需要两套完全不同的实现——MSDF 粗体把阈值从 0.5 降到 0.45 来扩展字形，位图粗体则采样相邻纹素。每个渲染特性都要实现两遍、测试两遍。

### 2. The CJK Compromise / CJK 的妥协

True MSDF requires vector glyph outlines. But PingFang SC (the best CJK font on macOS) locks its outlines inside Apple's TrueType Collection format. We had to fall back to bitmap→SDF conversion, which produces decent results but loses the sharp corner preservation that makes MSDF worthwhile. So we had three rendering qualities in one atlas: MSDF (great), SDF (okay), bitmap (limited). Not elegant.

真正的 MSDF 需要矢量字形轮廓。但苹方（macOS 上最好的中文字体）把轮廓锁在苹果的 TrueType Collection 格式里。我们只能退回位图→SDF 转换，效果还行但丢失了 MSDF 的尖角保留能力。于是一张图集里有三种渲染质量：MSDF（好）、SDF（凑合）、位图（有限）。不优雅。

### 3. Python Toolchain Dependency / Python 工具链依赖

Generating the atlas required: Python 3 + scipy + Pillow + the `msdfgen` C++ binary. For a Rust engine, this felt wrong. The generation script was 800+ lines of Python orchestrating multiple external tools.

生成图集需要：Python 3 + scipy + Pillow + `msdfgen` C++ 二进制。对于一个 Rust 引擎来说，这不对味。生成脚本是 800 多行 Python，调度多个外部工具。

### 4. The 8192×8192 Problem / 8192×8192 的问题

To fit all symbols (40960 sprites + 2560 TUI + 768 emoji + 4096 CJK) with their SDF padding, we needed an 8192×8192 atlas. That's 256MB uncompressed RGBA. Some mobile GPUs and older WebGL implementations don't support textures this large.

为了放下所有符号（40960 精灵 + 2560 TUI + 768 Emoji + 4096 CJK）加上 SDF 的边距，我们需要 8192×8192 的图集。未压缩 RGBA 就是 256MB。部分移动端 GPU 和旧版 WebGL 实现不支持这么大的纹理。

---

## The Mipmap Solution (v2.4+) / Mipmap 方案（v2.4+）

We stepped back and asked: what are we actually trying to solve? We need crisp symbols at different display sizes. SDF achieves this through math (distance field + smoothstep). But there's a much simpler approach: **just pre-render each symbol at multiple resolutions**.

我们退后一步问自己：我们到底在解决什么问题？我们需要不同显示尺寸下清晰的符号。SDF 通过数学实现（距离场 + smoothstep），但有个简单得多的方案：**直接把每个符号预渲染成多个分辨率**。

### Architecture / 架构

```
Texture2DArray (multiple 4096×4096 layers)
├── Layer 0: sprites (mip0)
├── Layer 1: sprites (mip1)
├── Layer 2: TUI characters (mip0, high-res)
├── Layer 3: TUI characters (mip1, mid-res)
├── Layer 4: TUI characters (mip2, low-res)
├── Layer 5-8: CJK (mip0/1/2)
├── Layer 9-10: Emoji (mip0/1/2)
└── ...
```

Each symbol has UV coordinates for all 3 mip levels:

每个符号有 3 级 mip 的 UV 坐标：

```rust
pub struct Tile {
    pub cell_w: u8,       // 1=normal, 2=wide (CJK/Emoji)
    pub cell_h: u8,       // 1=single, 2=tall (TUI/CJK)
    pub is_emoji: bool,   // Color emoji (no tint modulation)
    pub mips: [MipUV; 3], // 3 mipmap levels
}
```

The engine picks the right mip at render time based on actual pixel size:

引擎在渲染时根据实际像素尺寸选择合适的 mip：

| Condition / 条件 | Mip Level | Use Case / 场景 |
|-------------------|-----------|-----------------|
| `per_unit >= 48` | mip0 (high) | Fullscreen, 4K / 全屏 4K |
| `per_unit >= 24` | mip1 (mid) | Normal window / 普通窗口 |
| `per_unit < 24` | mip2 (low) | Small window / 小窗口 |

### What We Gained / 收获了什么

**1. One rendering path.** No more MSDF/bitmap branching. Every symbol is a bitmap at the right resolution. The fragment shader is dead simple:

**1. 单一渲染路径。** 不再有 MSDF/位图分支。每个符号都是正确分辨率的位图。片段着色器极其简单：

```wgsl
let texColor = textureSample(t_texture, s_sampler, uv, layer);
output.color = texColor * instance_color;
```

That's it. No `median3`, no `smoothstep`, no `fwidth`. Same code path for sprites, TUI, CJK, and emoji.

就这样。没有 `median3`，没有 `smoothstep`，没有 `fwidth`。精灵、TUI、CJK、Emoji 走同一条代码路径。

**2. Better CJK quality.** Pre-rendered PingFang SC at high resolution looks better than bitmap-to-SDF. We're using the native font renderer (macOS CoreText) at the target resolution instead of mathematically approximating edges.

**2. 更好的 CJK 质量。** 高分辨率预渲染的苹方比位图转 SDF 更好看。我们用原生字体渲染器（macOS CoreText）在目标分辨率直接渲染，而不是用数学近似边缘。

**3. Pure Rust toolchain.** The atlas generator is now a Rust tool (`cargo pixel symbols`), using CoreText bindings directly. No Python, no scipy, no external `msdfgen` binary.

**3. 纯 Rust 工具链。** 图集生成器现在是 Rust 工具（`cargo pixel symbols`），直接用 CoreText 绑定。没有 Python，没有 scipy，没有外部 `msdfgen` 二进制。

**4. Smaller max texture.** 4096×4096 layers instead of one 8192×8192 atlas. More compatible across devices, and the array format means we can add layers without increasing individual texture dimensions.

**4. 更小的最大纹理。** 4096×4096 的层而不是一张 8192×8192 的图集。设备兼容性更好，数组格式意味着可以加层而不增加单张纹理尺寸。

**5. DP shelf-packing.** The tool uses dynamic programming to optimally pack symbols into layers, minimizing wasted space.

**5. DP 货架打包。** 工具用动态规划把符号最优地打包进各层，最小化浪费空间。

### What We Lost / 失去了什么

**Infinite scalability.** SDF gives you mathematically perfect edges at *any* scale. With 3 mip levels, we have 3 fixed resolutions. If someone runs MDPT on an 8K display... well, mip0 at 48+ pixels per unit is still pretty good, but it's not mathematically perfect.

**无限可伸缩性。** SDF 在*任何*缩放下都能给出数学上完美的边缘。有了 3 级 mip，我们有 3 个固定分辨率。如果有人在 8K 显示器上运行 MDPT……嗯，mip0 在 48+ 像素/单位下仍然很好，但不是数学上完美。

**Memory efficiency at extreme scales.** A single SDF texture works at all scales. Mipmaps store 3 copies of every symbol. In practice, the total is similar because SDF needs larger cells for the distance field padding, and our layers are more tightly packed.

**极端缩放下的内存效率。** 一张 SDF 纹理在所有缩放下都能用。Mipmap 存了每个符号的 3 个副本。但实际上总量差不多，因为 SDF 需要更大的格子来放距离场边距，而我们的层打包得更紧密。

---

## The Uncomfortable Truth / 不太舒服的真相

SDF/MSDF is a beautiful technique. The math is elegant. The demo looks impressive. But in production, for our use case:

SDF/MSDF 是一项漂亮的技术。数学优雅，Demo 看着很震撼。但在生产环境中，对于我们的场景：

- We spent more time debugging the SDF pipeline than writing actual features
- The "works at any scale" promise was undermined by the CJK fallback to bitmap-SDF
- The Python toolchain was a constant friction point for contributors

- 我们花在调试 SDF 管线上的时间比写实际功能还多
- "任意缩放都能用"的承诺被 CJK 退回位图-SDF 削弱了
- Python 工具链对贡献者来说是持续的摩擦点

The mipmap approach is "boring" engineering. No elegant math, no cool shader tricks. Just pre-render at multiple sizes and pick the right one. But it works reliably, the code is simple, and we haven't had a single rendering bug since the switch.

Mipmap 方案是"无聊"的工程。没有优雅的数学，没有酷炫的着色器技巧。只是预渲染多个尺寸然后选对的那个。但它可靠地工作，代码简单，切换以来没有一个渲染 Bug。

Sometimes the best technical decision is the least interesting one.

有时候最好的技术决策是最不有趣的那个。

## Numbers / 数据对比

| | SDF/MSDF (v2.3) | Mipmap (v2.4+) |
|---|---|---|
| Atlas size / 图集尺寸 | 8192×8192 (single) | 4096×4096 × ~15 layers |
| Shader complexity / 着色器复杂度 | Dual path + median3 | Single path |
| Toolchain / 工具链 | Python + scipy + msdfgen | Pure Rust |
| Draw calls | 1 | 1 |
| CJK quality / CJK 质量 | Acceptable (bitmap-SDF) | Good (native rendering) |
| Scaling / 缩放 | Infinite (math) | 3 discrete levels |
| Code touched / 涉及代码 | shader + render + Python tool | shader + render + Rust tool |
| Bold impl / 粗体实现 | 2 paths | 1 path |

---

## Takeaways / 总结

1. **SDF is not always the answer for text rendering.** It shines when you need truly infinite scalability with minimal memory. For a game engine with known display size ranges, discrete mipmaps are simpler and good enough.

2. **Toolchain matters.** Replacing 800 lines of Python + external C++ binaries with a native Rust tool made the project more maintainable and contributor-friendly.

3. **Uniformity beats cleverness.** One rendering path for all symbol types is easier to optimize, debug, and extend than a clever dual-path shader.

4. **Know your actual requirements.** We needed "crisp at 720p through 4K." We didn't need "mathematically perfect at 16K." Three mip levels cover our actual use cases perfectly.

---

1. **SDF 并非文字渲染的万能解。** 它适用于需要真正无限可伸缩性且内存最小化的场景。对于显示尺寸范围已知的游戏引擎，离散 mipmap 更简单且足够好。

2. **工具链很重要。** 用原生 Rust 工具替换 800 行 Python + 外部 C++ 二进制，让项目更易维护、对贡献者更友好。

3. **统一胜过聪明。** 所有符号类型走一条渲染路径，比聪明的双路径着色器更容易优化、调试和扩展。

4. **了解你的实际需求。** 我们需要"720p 到 4K 都清晰"，不需要"16K 下数学上完美"。三级 mip 完美覆盖了我们的实际场景。
